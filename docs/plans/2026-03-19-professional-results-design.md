# Professional Results Design

**Date:** 2026-03-19
**Scope:** 结果专业感升级、付费比较报告、支付接入、日志观测
**Goal:** 把问名从“有文气的起名工具”升级成“可收费的命名内容产品”，并以结果页成交为核心。

## Problem

当前产品已经有首页、测试、起名、评分、收藏、分享的完整骨架，但还不具备收费能力，主要问题有四类：

1. 结果不够值钱
   现有生成和评分结果更像工具输出，不像可付费交付物。用户看完后很难产生“这值得花 9.9-29 元解锁完整版”的感觉。

2. UI 气质和交付物不匹配
   页面有文气，但缺乏“专业作品感”。视觉表达和结构表达都不足以支撑低客单价内容商品。

3. 分享链路弱于成交链路
   现有分享主要是测试结果分享给伴侣，适合传播，但不直接服务付费转化。

4. Prompt 维度偏薄
   当前 prompt 主要关注风格和硬约束，没有充分约束惊喜度、候选差异、比较判断、专业解释结构、传统维度等。

## Product Decision

### Positioning

问名本轮定位为：

- 可传播、可收费的命名内容产品
- 免费靠“先测方向，再起名”拉新
- 成交靠“结果页升级为专业比较报告”

### Core User

本轮优先服务用户：

1. 已经有 2-3 个候选名，想做选择的人
2. 已经有现成名字，想验证好不好的人
3. 还没有候选名、先通过测试和起名获取方向的人

### Paid Offer

本轮付费物定义为：

- `2-3 个候选名的横向比较 + 推荐结论 + 传统维度模块`

不是：

- 单纯“再多给几个名字”
- 重服务型高客单咨询
- 会员订阅

价格带：

- `9.9-29`

### Language Direction

语言方向采用：

- 主体气质偏“高级内容产品”
- 核心判断口吻偏“专业命名顾问”
- 局部语感保留一点书卷气

明确约束：

- 收藏相关功能名不再用“书”隐喻
- `藏书阁` 从核心功能词中移除
- 强书卷隐喻如 `卷宗`、`研墨` 不再作为关键动作文案

## Approaches

### Option A: 只做 UI 提升

- 重绘首页、结果页、分享页
- 不改变付费模型和结果结构

**Pros**

- 速度快
- 视觉改善明显

**Cons**

- 不能真正支撑收费
- 结果仍像普通工具输出

### Option B: 只做结果与 prompt 升级

- 重做 prompt
- 重写结果结构
- 先不接支付

**Pros**

- 直接打击“结果不够惊艳”
- 为后续收费打底

**Cons**

- 没有真实成交闭环
- 很难验证商业价值

### Option C: 测试拉新 + 专业结果页 + 付费比较报告 + 支付 + 日志

- 首页继续主打测试
- 结果页升级为可阅读、可升级的专业交付物
- 接入一次性支付，解锁完整比较报告
- 补齐日志与漏斗观测

**Pros**

- 同时覆盖拉新、成交、验证
- 能真实验证是否有人愿意付费
- 结果页和商业化架构一次到位

**Cons**

- 改动面最大
- 需要新增支付状态和内容验权

**Decision**

- 采用 `Option C`

## Information Architecture

### Free Funnel

1. 用户进入首页，主 CTA 仍是 `开始测试`
2. 测试完成后进入起名或评分链路
3. 免费结果页展示摘要版结果
4. 用户在结果页点击升级，购买完整比较报告

### Paid Funnel

1. 用户在结果页选择 2-3 个候选名
2. 前端请求创建结账会话
3. 跳转到 Lemon Squeezy 托管结账页
4. 支付成功后回跳
5. Worker 验证支付状态
6. 返回完整报告

### Sharing Funnel

1. 免费分享物优先是 `单个名字评分/解读结果卡`
2. 分享落地页承接回：
   - 评分
   - 测试
   - 付费升级
3. 完整付费报告默认不公开分享全文，只分享摘要和 CTA

## UX Decisions

### Homepage

- 首页主任务保持 `先测方向，再起名`
- 同时增加更清楚的次入口，承接“已有名字/候选”的用户
- 视觉目标不是“更花”，而是“更像能收费的产品”

### Results

免费结果页要具备三层结构：

1. 结论层
   - 当前推荐倾向
   - 一句话判断

2. 专业解释层
   - 名字为什么成立
   - 文学/意境分析
   - 风险与边界提示

3. 升级层
   - 明确告诉用户付费后会获得什么额外判断

### Paid Report

完整付费报告必须包含：

- 总推荐结论
- 候选名逐一点评
- 横向比较
- 推荐排序
- 为什么 A 比 B 更成立
- 传统维度模块

这份报告的体验目标是“作品感”，而不是“解锁更多字段”。

## Prompt Strategy

### Generation Prompt

新增维度：

- 候选间风格差异
- 惊喜度约束
- 候选名成立理由
- 家庭沟通可解释性
- 避免同质化

### Score / Compare Prompt

新增维度：

- 逐名成立理由
- 横向判断标准
- 最终推荐结论
- 风险提示
- 传统维度补充

约束：

- 免费结果输出摘要版结构
- 付费结果输出完整版结构
- 不能让前端先拿到全文后再隐藏

## Payment Decision

### Provider

第一阶段使用 `Lemon Squeezy`

原因：

- 更适合个人开发者快速验证一次性数字交付
- 托管结账页可直接落地
- 本身是 Merchant of Record，降低税务和合规负担
- 支持 Alipay / WeChat Pay 等支付方式

### Why Not Direct WeChat / Alipay First

直接接微信/支付宝官方支付需要更重的主体资质和支付场景审核。对于当前阶段，最大问题不是“有没有支付通道”，而是“能否先验证付费意愿”。因此先用 MoR，后续再决定是否补国内直连。

## Logging And Analytics

### Goal

本轮必须让你能回答三类问题：

1. 用户有没有走到结果页
2. 用户有没有点击升级
3. 用户有没有真的付费成功

### Two-Layer Logging

#### Layer 1: Debug Logs

用途：

- API 报错
- webhook 回调失败
- 支付状态校验失败

实现：

- Worker 中保留 `console.log` / `console.error`
- 线上通过 Cloudflare Workers Logs 查看
- 开发和线上排查可用 `wrangler tail`

#### Layer 2: Product Event Logs

用途：

- 记录漏斗和用户行为
- 统计测试完成率、升级点击率、支付成功率、分享率

实现建议：

- 新增一个结构化事件表，首选 D1
- 每条事件包含：
  - `event_id`
  - `session_id`
  - `event_name`
  - `page`
  - `payload_json`
  - `created_at`

为什么不用只看 console：

- console 适合排错，不适合做产品分析
- 事件日志需要持久化和可查询

### How To View Logs

调试日志查看方式：

- Cloudflare Dashboard 的 Workers Logs
- `wrangler tail`

用户事件查看方式：

- 开发阶段：`wrangler d1 execute` 直接查表
- 运营阶段：做一个简单的管理页或导出脚本查看

## Technical Architecture

### New Server Responsibilities

Worker 新增职责：

- 创建 Lemon Squeezy checkout
- 接收并验证 webhook
- 管理报告摘要与全文的访问权限
- 写入结构化事件日志

### Data Boundary

免费结果和付费结果边界必须在服务端控制：

- 摘要版可直接返回前端
- 完整版只在支付成功后返回

### Persistence

建议新增：

- D1：支付记录、报告状态、事件日志

不建议：

- 只靠 localStorage 管付费状态
- 只靠前端参数判断已支付

## Acceptance Mapping

完成后应满足：

1. 首页仍以 `开始测试` 为主 CTA，但次级入口能承接已有候选用户
2. 结果页具备明显专业感，足以支撑低客单价付费升级
3. 收藏相关命名移除 `藏书阁`
4. 免费结果和付费结果边界清楚，完整报告必须服务端验权
5. Lemon Squeezy 结账、回跳、webhook、解锁完整报告链路打通
6. 用户行为和支付链路有结构化日志，且你知道怎么查看
